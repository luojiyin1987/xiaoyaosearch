# 架构决策 - MCP 服务器采用 FastAPI 集成方案（fastmcp 框架）

**决策日期**：2026年03月11日
**最后更新**：2026年03月12日（采用 fastmcp 框架）

## 背景

在为小遥搜索添加 MCP (Model Context Protocol) 服务器能力时，面临两种架构选择：

1. **独立进程方案**：MCP 服务器作为独立进程运行，通过 stdio 与 Claude Desktop 通信
2. **集成方案**：将 MCP 服务器集成到现有 FastAPI 进程中，通过 SSE 端点与 Claude Desktop 通信

独立进程方案存在以下问题：
- **内存浪费**：两个进程各自加载 AI 模型（BGE-M3、FasterWhisper、CN-CLIP），内存占用翻倍（8-12GB vs 4-6GB）
- **模型重复加载**：启动时间延长，资源利用率低
- **进程间通信复杂**：MCP 进程无法直接访问 FastAPI 进程中的模型对象
- **部署复杂**：需要管理两个独立进程的生命周期

### 框架选择（2026-03-12 更新）

在确定采用集成方案后，进一步面临 MCP Python 框架的选择：

1. **mcp-python-sdk**：Anthropic 官方 Python SDK，需要手动实现 SSE 传输层
2. **fastmcp**：PrefectHQ 维护的 Python MCP 框架，内置 SSE 支持，使用装饰器模式

**最终选择**：采用 **fastmcp** 框架

选择理由：
- **代码简洁**：装饰器模式 (`@mcp.tool`)，代码量减少约 60%
- **自动化程度高**：自动 Schema 生成、参数验证、SSE 处理
- **FastAPI 原生集成**：`mcp.http_app(path)` 一行代码挂载
- **学习曲线平缓**：API 简单直观，开发效率高
- **活跃维护**：PrefectHQ 社区活跃，问题响应及时

## 解决方案

**采用集成方案**：将 MCP 服务器集成到现有 FastAPI 进程中，添加 SSE (Server-Sent Events) 传输端点，使用 **fastmcp** 框架实现。

### 具体实现

1. **集成架构（fastmcp）**
   - 在 FastAPI 的 `lifespan` 生命周期中初始化 FastMCP 服务器
   - 通过 `mcp.http_app(path="/mcp/sse")` 获取 ASGI 应用
   - 使用 `app.mount("/mcp", mcp_asgi_app)` 挂载到 FastAPI
   - 工具函数直接调用现有搜索服务（无需适配器）

2. **通信方式**
   ```
   Claude Desktop → HTTP SSE → FastAPI (/mcp/sse) → FastMCP → 现有搜索服务
   ```

3. **Claude Desktop 配置**
   ```json
   {
     "mcpServers": {
       "xiaoyao-search": {
         "url": "http://127.0.0.1:8000/mcp/sse"
       }
     }
   }
   ```

4. **代码结构（fastmcp 简化版）**
   ```
   backend/app/mcp/
   ├── __init__.py
   ├── server.py              # FastMCP 服务器实例
   ├── config.py              # MCP 配置管理
   └── tools/                 # 工具模块
       ├── semantic_search.py  # @mcp.tool 装饰器
       ├── fulltext_search.py  # @mcp.tool 装饰器
       ├── voice_search.py     # @mcp.tool 装饰器
       ├── image_search.py     # @mcp.tool 装饰器
       └── hybrid_search.py    # @mcp.tool 装饰器
   ```

5. **依赖更新**
   ```bash
   # backend/requirements.txt
   fastmcp>=0.4.0
   ```

### fastmcp 核心优势

| 特性 | mcp-python-sdk | fastmcp |
|------|----------------|---------|
| 工具定义 | 继承 `BaseTool` 类 | `@mcp.tool` 装饰器 |
| Schema 生成 | 手动定义 `inputSchema` | 自动从类型提示生成 |
| SSE 支持 | 需手动实现传输层 | 内置 `http_app()` 方法 |
| FastAPI 集成 | 需自定义端点 | `app.mount()` 一行挂载 |
| 参数验证 | 手写验证逻辑 | Pydantic 自动验证 |
| 代码量 | ~600 行 | ~200 行（减少 60%） |

## 影响

### 优势

1. **内存高效**：AI 模型只加载一次，节省 4-6GB 内存（内存增量 ≤ 100MB）
2. **部署简单**：单一进程，无需管理多个服务
3. **共享服务**：Electron 前端和 Claude Desktop 共享相同的搜索服务和模型
4. **直接调用**：MCP 工具直接调用搜索服务，无 HTTP 开销
5. **向后兼容**：现有功能和 API 完全不受影响
6. **代码简洁**（fastmcp）：装饰器模式，代码量减少 60%
7. **开发效率高**（fastmcp）：自动 Schema 生成，开发时间缩短 40%

### 劣势

1. **职责耦合**：FastAPI 同时服务两种客户端（Electron + Claude Desktop）
2. **故障影响**：MCP 相关问题可能影响主服务（概率极低）
3. **扩展性受限**：未来如需独立扩展 MCP 服务需要重构
4. **第三方依赖**（fastmcp）：fastmcp 由 PrefectHQ 维护，非 Anthropic 官方（但社区活跃）

### 依赖关系

- 依赖现有 FastAPI 服务的生命周期管理
- 依赖现有 AI 模型和搜索服务的初始化
- 需要添加 `fastmcp>=0.4.0` Python 包依赖
- 依赖 PrefectHQ 社区对 fastmcp 的持续维护

### 兼容性

- 完全向后兼容，不影响现有 Electron 前端功能
- MCP 功能可通过 `MCP_SSE_ENABLED` 环境变量禁用
