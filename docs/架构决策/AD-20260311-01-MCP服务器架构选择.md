# 架构决策 - MCP 服务器采用 FastAPI 集成方案

**决策日期**：2026年03月11日

## 背景

在为小遥搜索添加 MCP (Model Context Protocol) 服务器能力时，面临两种架构选择：

1. **独立进程方案**：MCP 服务器作为独立进程运行，通过 stdio 与 Claude Desktop 通信
2. **集成方案**：将 MCP 服务器集成到现有 FastAPI 进程中，通过 SSE 端点与 Claude Desktop 通信

独立进程方案存在以下问题：
- **内存浪费**：两个进程各自加载 AI 模型（BGE-M3、FasterWhisper、CN-CLIP），内存占用翻倍（8-12GB vs 4-6GB）
- **模型重复加载**：启动时间延长，资源利用率低
- **进程间通信复杂**：MCP 进程无法直接访问 FastAPI 进程中的模型对象
- **部署复杂**：需要管理两个独立进程的生命周期

## 解决方案

**采用集成方案**：将 MCP 服务器集成到现有 FastAPI 进程中，添加 SSE (Server-Sent Events) 传输端点。

### 具体实现

1. **集成架构**
   - 在 FastAPI 的 `lifespan` 生命周期中初始化 MCP 服务器
   - 添加 `/mcp/sse` 端点用于 Claude Desktop 连接
   - 通过 `SearchAdapter` 复用现有搜索服务

2. **通信方式**
   ```
   Claude Desktop → HTTP SSE → FastAPI (/mcp/sse) → MCP Server → SearchAdapter → 现有搜索服务
   ```

3. **Claude Desktop 配置**
   ```json
   {
     "mcpServers": {
       "xiaoyao-search": {
         "url": "http://127.0.0.1:8000/mcp/sse"
       }
     }
   }
   ```

4. **代码结构**
   - 新增 `backend/app/mcp/` 模块
   - 修改 `backend/main.py` 集成 MCP 服务器
   - 无需修改现有搜索服务

## 影响

### 优势

1. **内存高效**：AI 模型只加载一次，节省 4-6GB 内存（内存增量 ≤ 100MB）
2. **部署简单**：单一进程，无需管理多个服务
3. **共享服务**：Electron 前端和 Claude Desktop 共享相同的搜索服务和模型
4. **直接调用**：MCP 工具直接调用搜索服务，无 HTTP 开销
5. **向后兼容**：现有功能和 API 完全不受影响

### 劣势

1. **职责耦合**：FastAPI 同时服务两种客户端（Electron + Claude Desktop）
2. **故障影响**：MCP 相关问题可能影响主服务（概率极低）
3. **扩展性受限**：未来如需独立扩展 MCP 服务需要重构

### 依赖关系

- 依赖现有 FastAPI 服务的生命周期管理
- 依赖现有 AI 模型和搜索服务的初始化
- 需要添加 `mcp` Python 包依赖

### 兼容性

- 完全向后兼容，不影响现有 Electron 前端功能
- MCP 功能可通过 `MCP_SSE_ENABLED` 环境变量禁用
