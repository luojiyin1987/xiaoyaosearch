# 架构决策 - Docker部署方案：本地与Docker双模式架构

**决策日期**：2026年02月03日

## 背景

小遥搜索作为本地桌面应用，当前采用Electron前端+本地Python后端的部署方式。为了满足以下需求：

1. **简化用户安装部署**：降低环境配置复杂度，减少依赖冲突
2. **支持不同用户群体**：
   - 个人用户：期望开箱即用的桌面体验
   - 开发者/技术用户：希望有更灵活的部署选项
3. **支持CPU和GPU加速**：满足不同硬件配置用户的性能需求
4. **解决本地路径访问问题**：Electron选择本地文件夹时，Docker容器无法直接访问

需要设计一个既能保持本地应用体验，又能提供Docker部署优势的方案。

## 解决方案

**采用方案C：本地部署 + Docker可选双模式**

### 部署模式对比

| 特性 | 本地部署模式 | Docker部署模式 |
|------|-------------|----------------|
| **目标用户** | 个人用户 | 开发者/技术用户 |
| **安装复杂度** | 中等（需Python环境） | 低（需Docker环境） |
| **文件访问** | 直接访问任意本地路径 | 访问预配置挂载目录 |
| **环境隔离** | 无 | 完全隔离 |
| **更新维护** | 手动更新依赖 | 镜像版本管理 |
| **GPU支持** | 需手动配置CUDA | 通过NVIDIA Container Toolkit |
| **适用场景** | 日常桌面使用 | 开发/测试/服务器部署 |

### 技术实现要点

#### 1. Docker镜像设计

**基础镜像策略**：
- 使用`python:3.10-slim`作为基础镜像
- 通过构建参数控制CPU/GPU版本
- 使用多阶段构建优化镜像大小

**CPU/GPU支持**：
```dockerfile
# CPU版本（默认）
FROM python:3.10-slim
RUN pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cpu
RUN pip install faiss-cpu

# GPU版本（通过构建参数）
ARG GPU=false
RUN if [ "$GPU" = "true" ]; then \
    pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cu121; \
    else \
    pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cpu; \
    fi
```

#### 2. 数据持久化方案

**Docker部署的数据目录结构**：
```
xiaoyaosearch/
├── data/                          # 数据目录（挂载到容器）
│   ├── database/                  # SQLite数据库
│   ├── indexes/                   # 索引文件
│   │   ├── faiss/                # 向量索引
│   │   └── whoosh/               # 全文索引
│   ├── configs/                   # 配置文件
│   ├── logs/                      # 日志文件
│   └── files/                     # 待索引文件目录（用户放置文件）
├── backend/                       # 后端代码
└── docker-compose.yml            # Docker编排配置
```

#### 3. 前端配置适配

通过环境变量控制后端连接：
```typescript
// 前端检测部署模式
const BACKEND_MODE = import.meta.env.VITE_BACKEND_MODE || 'local'

const API_BASE_URL = BACKEND_MODE === 'docker'
  ? 'http://localhost:8000'  // Docker容器端口
  : 'http://127.0.0.1:8000'  // 本地后端
```

#### 4. 用户引导流程

**首次启动引导**：
1. 检测部署模式（本地/Docker）
2. Docker模式：引导用户配置文件目录
3. 提供部署模式切换选项（需重启应用）

### 使用场景分配

**本地部署模式（推荐）**：
- 个人日常使用
- 需要搜索多个不同位置的文件
- 对Docker不熟悉的用户
- 需要频繁切换文件目录

**Docker部署模式**：
- 开发环境测试
- 服务器部署场景
- 需要快速部署/回滚
- 希望环境完全隔离
- 需要GPU加速但不想配置CUDA环境

## 影响

### 正面影响

1. **降低部署门槛**：Docker版本无需配置Python环境
2. **灵活适配**：用户可根据需求选择合适的部署方式
3. **开发友好**：开发者可使用Docker快速搭建测试环境
4. **环境一致性**：Docker版本保证跨平台一致性
5. **GPU支持简化**：通过NVIDIA Container Toolkit简化GPU配置

### 需要注意

1. **维护成本增加**：需要维护两套部署方式的文档和脚本
2. **Docker学习曲线**：普通用户可能需要学习基本Docker命令
3. **文件访问限制**：Docker模式需要用户将文件放入指定目录
4. **资源占用**：Docker相比本地部署有额外的资源开销
5. **Windows性能**：Windows上Docker性能可能略低于原生

### 依赖关系

1. 需要创建Docker相关配置文件（Dockerfile、docker-compose.yml）
2. 需要编写Docker部署文档
3. 前端需要增加部署模式检测和配置
4. 需要提供部署模式切换的用户界面

### 后续工作

1. **高优先级**：
   - 创建Dockerfile（CPU/GPU版本）
   - 创建docker-compose.yml
   - 编写Docker部署文档

2. **中优先级**：
   - 前端增加部署模式配置
   - 创建一键启动脚本
   - 添加部署模式切换UI

3. **低优先级**：
   - 提供预构建镜像（Docker Hub）
   - CI/CD集成自动构建
   - 健康检查和监控集成
