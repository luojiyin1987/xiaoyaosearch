# 架构决策 - 云端嵌入模型与索引重建方案

**决策日期**：2026年3月26日

## 背景

小遥搜索当前使用本地 BGE-M3 嵌入模型（1024维向量）进行语义搜索。随着用户对多语言支持和更高质量嵌入的需求增加，需要添加云端嵌入模型支持。

**当前痛点**：
- 本地 BGE-M3 模型对英文等其他语言的支持较弱
- 云端嵌入模型（如 OpenAI text-embedding-3-small）提供更高质量的向量和多语言支持
- 不同嵌入模型的向量空间不兼容，直接切换会导致搜索失效

**核心挑战**：
- 不同嵌入模型的向量维度不同（BGE-M3: 1024维，OpenAI: 1536维）
- 即使维度相同，向量空间也不同，无法直接复用索引
- 切换模型需要重建整个 Faiss 索引，耗时较长

**目标**：
- 支持云端嵌入模型，提供更高质量的语义搜索
- 保证模型切换时索引的正确性和一致性
- 最小化切换期间的用户体验影响

## 解决方案

### 核心决策一：使用模型原始维度

**决策**：不进行向量维度归一化，直接使用模型的原始向量维度。

**理由**：
1. **保持精度**：截断、填充或投影都会损失向量精度，降低搜索质量
2. **简化实现**：无需处理维度转换逻辑，降低复杂度
3. **性能最优**：直接使用原始向量，避免额外计算开销

**影响**：
- 不同模型的索引结构不同，必须分别存储
- 切换模型时必须重建索引，无法复用

### 核心决策二：索引重建策略（方案B）

**决策**：采用独立端点 + 内存状态管理的索引重建方案。

**方案对比**：

| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| A. 混用现有索引任务系统 | 复用代码，改动小 | 与现有索引任务耦合复杂 | ❌ |
| **B. 独立端点 + 内存状态** | 完全隔离，实现简单，风险低 | 需要新增端点 | ✅ |
| C. 后台队列系统 | 解耦彻底，可扩展 | 架构复杂，开发周期长 | ❌ |

**方案B架构设计**：

```
┌─────────────────────────────────────────────────────────────┐
│                      前端界面                                 │
├─────────────────────────────────────────────────────────────┤
│  1. 用户切换嵌入模型（本地 → 云端）                           │
│  2. 弹出确认对话框："切换需要重建索引，预计耗时 X 分钟"        │
│  3. 用户确认后，调用 POST /api/index/rebuild/start           │
└────────────────────┬────────────────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────────────────┐
│              独立重建服务（内存状态）                          │
├─────────────────────────────────────────────────────────────┤
│  IndexRebuildService                                         │
│  - current_task: Optional[RebuildTask]  # 内存存储，不写DB   │
│  - _lock: asyncio.Lock                                   │
│                                                              │
│  状态流程：                                                   │
│  1. start_rebuild() → 备份现有索引 → 切换模型 → 后台执行       │
│  2. _execute_rebuild() → 逐文件生成新向量 → 更新 Faiss 索引    │
│  3. get_status() → 返回进度、ETA                             │
│  4. cancel_rebuild() → 取消任务 → 恢复原模型                  │
└────────────────────┬────────────────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────────────────┐
│                   API 端点（独立命名空间）                     │
├─────────────────────────────────────────────────────────────┤
│  POST   /api/index/rebuild/start   # 开始重建                │
│  GET    /api/index/rebuild/status  # 查询进度                │
│  POST   /api/index/rebuild/cancel  # 取消重建                │
│                                                              │
│  注意：不与现有 /api/index/* 端点冲突                         │
└─────────────────────────────────────────────────────────────┘
```

### 核心决策三：类型互斥配置

**决策**：用户同时只能激活一种嵌入模型（本地或云端）。

**实现方式**：
1. 前端配置表单互斥显示（v-if 切换）
2. 后端激活新模型时自动将旧模型设为不活跃
3. 数据库 `is_active` 字段保证唯一性（应用层约束）

**配置结构**：

```json
// 本地 BGE-M3 配置
{
  "model_type": "embedding",
  "provider": "local",
  "model_name": "BAAI/bge-m3",
  "config": {
    "device": "cpu"
  },
  "is_active": true
}

// 云端 OpenAI 配置
{
  "model_type": "embedding",
  "provider": "cloud",
  "model_name": "text-embedding-3-small",
  "config": {
    "api_key": "sk-xxxxx",
    "endpoint": "https://api.openai.com/v1",
    "model": "text-embedding-3-small",
    "timeout": 30,
    "batch_size": 100
  },
  "is_active": false  // 激活云端时，本地自动变为 false
}
```

### 核心决策四：索引备份与回滚

**决策**：重建前自动备份索引，失败或取消时自动回滚。

**备份策略**：
- 备份位置：`data/index.backup/{timestamp}/`
- 备份内容：完整的 Faiss 索引文件
- 备份方式：`shutil.copytree()` 递归复制

**回滚机制**：
```
1. 用户取消重建 → 立即停止 → 恢复原模型配置
2. 重建失败 → 自动捕获异常 → 恢复原模型配置
3. 用户刷新页面 → 任务状态丢失 → 可重新查询状态

原模型配置包含：
- provider 类型
- model_name
- config 配置
```

| `backend/app/services/index_rebuild_service.py` | 新建 | 独立索引重建服务 |
| `backend/app/api/index_rebuild.py` | 新建 | 重建 API 端点 |
| `backend/app/services/ai_model_manager.py` | 修改 | 支持云端嵌入模型 |
| `backend/app/locales/zh_CN.json` | 修改 | 添加后端国际化 |
| `backend/app/locales/en_US.json` | 修改 | 添加后端国际化 |

| `frontend/src/locale/lang/en-us.ts` | 修改 | 添加前端国际化 |

### 数据库变更

**无需迁移**：现有 `ai_models` 表完全支持新增功能。

- `provider` 字段已支持 `local/cloud` 值
- `config_json` 字段以 JSON 格式灵活存储配置
- `is_active` 字段通过应用层约束实现互斥

### 性能影响

| 场景 | 影响 | 说明 |
|------|------|------|
| 首次配置云端模型 | 无 | 仅测试连接，不重建索引 |
| 切换嵌入模型 | 显著 | 需要重建整个 Faiss 索引 |
| 搜索性能 | 无差异 | 本地/云端向量搜索速度相同 |
| 内存占用 | 无变化 | 云端模型不占用模型内存 |

### 用户体验流程

```
1. 用户在配置页面选择云端嵌入模型
   ↓
2. 系统提示："切换需要重建索引，预计耗时 X 分钟，是否继续？"
   ↓
3. 用户确认 → 开始重建
   ↓
4. 显示重建进度（处理文件数 / 总文件数）
   ↓
5. 重建完成 → 自动切换到新模型
   ↓
6. 用户可立即使用新模型进行搜索
```

**中断处理**：
- 重建期间用户可以继续搜索（使用旧模型）
- 用户可以取消重建 → 自动回滚到原模型
- 重建失败 → 自动回滚到原模型 + 错误提示

### 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 重建时间过长 | 中 | 显示进度和 ETA，支持取消 |
| 重建期间网络中断 | 中 | 自动回滚，保留原索引 |
| 云端 API 调用失败 | 低 | 重试机制（tenacity），错误提示 |
| API 费用超预期 | 低 | 显示文件数量，用户主动确认 |
| 重建中断电 | 低 | 重建期间暂不删除备份，下次启动可恢复 |

### 依赖关系

```
后端服务实现 → 索引重建服务 → 前端 UI 改造 → 测试验证
   (6小时)        (4小时)         (4小时)        (6小时)
```

## 后续扩展

本架构支持以下功能的快速开发：

1. **多模型并存**：扩展为支持同时使用多个嵌入模型（A/B 测试）
2. **增量索引更新**：仅对新文件重建索引，减少切换耗时
3. **索引版本管理**：保留多个版本的索引，支持快速回滚
4. **混合搜索**：本地模型快速筛选 + 云端模型精排
