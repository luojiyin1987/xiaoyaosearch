# 架构决策 - 索引构建进度显示优化

**决策日期**：2026年2月24日

## 背景

当前索引构建功能存在进度显示bug：当文件扫描和处理完成后，前端进度条立即显示100%，但实际上后续的"保存数据库"和"构建索引"阶段仍在进行，导致用户误以为索引已完成。

**问题根源**：

当前进度计算公式（在 `IndexJobModel.to_dict()` 中）：
```python
progress = int((processed_files / total_files) * 100)
```

索引构建实际包含4个阶段：
1. **扫描文件**（0-30%）
2. **处理文件**（30-80%）- 当此阶段完成时 `processed_files = total_files`，进度显示100%
3. **保存数据库**（70-85%）
4. **构建索引**（85-100%）- Faiss向量索引构建耗时较长

问题在于第2阶段完成后，`processed_files` 已等于 `total_files`，导致前端显示100%，但实际还有20-30%的工作量。

## 解决方案

### 方案对比

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| 方案1 | 引入阶段概念，添加 `stage` 和 `stage_progress` 字段 | 语义清晰，进度准确 | 需要数据库迁移 |
| **方案2 ✅** | **延长 `processed_files` 语义，各阶段按权重更新** | **无需数据库迁移，实现快速** | **语义略有不准确** |
| 方案3 | 前端根据阶段计算进度 | 后端改动小 | 需前后端同步修改 |

### 最终决策：方案2

**核心思路**：将 `processed_files` 扩展为"虚拟进度单位"，各阶段按权重分配进度：

| 索引阶段 | 进度范围 | processed_files 计算公式 |
|---------|---------|------------------------|
| 扫描文件 | 0-10% | `total_files × 当前扫描进度 × 0.1` |
| 处理文件 | 10-70% | `total_files × 0.1 + (已处理数 / 总数) × 0.6` |
| 保存数据库 | 70-85% | `total_files × 0.7 + 保存进度 × 0.15` |
| 构建索引 | 85-99% | `total_files × 0.85 + 索引进度 × 0.14` |
| 完成 | 100% | `total_files` |

### 实施细节

**修改文件**：
- `backend/app/services/file_index_service.py` - 在各阶段更新进度

**关键修改点**：

1. **扫描阶段**（第327-352行）：不更新 processed_files，保持为0

2. **处理文件阶段**（第372-398行）：
```python
# 原逻辑：processed_count = i + 1
# 新逻辑：按10-70%权重计算
file_progress = 0.1 + ((i + 1) / len(all_files)) * 0.6
processed_count = int(total_files * file_progress)
```

3. **保存数据库阶段**（第414-421行）：
```python
# 保存完成后更新进度到 85%
processed_count = int(total_files * 0.85)
```

4. **构建索引阶段**（第423-456行）：
```python
# 索引开始时更新进度到 85%
# 索引完成后更新进度到 99%
```

5. **任务完成**（第458-465行）：
```python
# 设置为真正的 100%
processed_count = total_files
```

## 影响

### 技术影响

**代码变更**
- 修改文件：`file_index_service.py`（约5处进度更新逻辑）
- 无需数据库迁移
- 无需修改前端代码
- 无需修改API接口

**性能影响**
- 无性能影响，仅调整进度计算逻辑

### 兼容性影响

**向后兼容**
- ✅ 现有API保持兼容
- ✅ 现有数据库结构无需变更
- ✅ 前端代码无需修改

**用户体验影响**
- ✅ 进度条平滑增长，不会在80%时跳到100%
- ✅ 用户能准确感知索引构建进度
- ✅ 减少用户焦虑（不会误以为已完成但实际还在运行）

### 维护影响

**注意事项**
1. `processed_files` 字段语义已变化：不再代表真实处理的文件数，而是"进度单位"
2. 代码注释中需明确说明此字段的实际含义
3. 后续维护时需注意权重分配（10%-70%-85%-99%-100%）

**权重调整建议**
如果后续阶段耗时发生变化，可按以下方式调整权重：
- 扫描：5-10%（通常很快）
- 处理文件：50-60%（取决于文件数量和类型）
- 保存数据库：10-15%（通常较快）
- 构建索引：20-25%（取决于向量数量）

## 后续优化建议

虽然方案2可快速解决问题，但长期来看建议：

1. **考虑迁移到方案1**：当需要更精细的进度控制时，添加 `stage` 和 `stage_progress` 字段
2. **前端增强**：可考虑在前端显示当前阶段名称（如"正在构建索引..."）
3. **耗时统计**：记录各阶段实际耗时，用于优化权重分配
