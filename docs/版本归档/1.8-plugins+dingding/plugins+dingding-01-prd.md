## 1. 特性概述

### 1.1 定位（一句话）

为小遥搜索添加钉钉文档数据源识别能力，通过解析钉钉导出工具自动生成的 `.xyddjson` 元数据文件，实现零配置、自动识别的钉钉文档智能搜索体验。

### 1.2 背景与动机

- **用户痛点**：
  - 痛点1：团队文档存储在钉钉文档中，用户需要定期导出到本地进行搜索
  - 痛点2：导出的钉钉文档与本地文件混合存储，无法区分数据来源
  - 痛点3：搜索到钉钉文档后，无法快速跳转回钉钉原文链接

- **业务价值**：
  - 降低使用门槛：用户导出文档后系统自动识别，无需任何配置
  - 提升搜索体验：在搜索结果中清晰标识钉钉文档来源
  - 快速定位原文：支持从搜索结果直接跳转到钉钉文档
  - 性能优化：独立元数据文件，解析速度快于内容解析方案

- **技术必要性**：
  - 钉钉导出工具已完成开发，需配套元数据解析能力
  - `.xyddjson` 格式标准化，可复用到未来其他平台（企业微信、飞书等）
  - 基于插件化架构，复用数据源识别机制

- **与飞书的对比**：

| 对比项 | 飞书方案 | 钉钉方案 | 优势方 |
|-------|---------|---------|--------|
| 配置难度 | 零配置 | 零配置 | 平手 |
| 识别方式 | 解析文件内容 | 解析独立JSON文件 | 钉钉 |
| 解析性能 | 中等（~1ms） | 高（<1ms） | 钉钉 |
| 可靠性 | 依赖导出格式 | 标准JSON | 钉钉 |
| 扩展性 | 仅限飞书 | 可用于其他平台 | 钉钉 |

### 1.3 目标用户

- **主要用户**：
  - 使用钉钉文档进行团队协作的知识工作者
  - 需要将钉钉文档与本地文件一起搜索的用户
  - 已安装钉钉导出工具的用户

- **使用场景**：
  - 产品经理搜索本地产品文档 + 钉钉需求文档
  - 开发者搜索本地代码 + 钉钉技术方案文档
  - 用户搜索后通过原文链接快速定位钉钉文档

### 1.4 与主版本的关系

- **依赖关系**：
  - 依赖主PRD中的多模态搜索功能
  - 依赖主PRD中的索引构建服务
  - 依赖插件化架构（plugins+yuque特性）
  - 依赖钉钉导出工具（外部依赖）

- **影响范围**：
  - 后端新增钉钉数据源识别插件
  - 文件扫描服务增加 `.xyddjson` 元数据文件检测
  - 搜索结果支持钉钉数据源标识（前端已支持）

## 4. 核心流程（增量部分）

### 4.1 新增流程

**流程1：钉钉文档识别**
```
文件扫描 → 检测到 .xyddjson 文件 → 读取JSON元数据
                                      ↓
                              解析元数据（dentryUuid、url、exportTime）
                                      ↓
                              关联原文件 → 设置 source_type=dingding
                                      ↓
                              存储到数据库 → 构建索引
```

**流程2：元数据文件与原文件关联**
```
扫描目录 → 发现文件 → 检查是否存在同名 .xyddjson
                            ↓
                        存在元数据文件?
                       /   \
                      是    否
                     /       \
              读取元数据    标记为本地文件
                      \       /
                       \     /
                        ↓   /
                设置 source_type=dingding
                        ↓
                构建索引（包含原文链接）
```

### 4.2 修改流程

**修改：文件扫描流程**
```
原流程：扫描文件 → 解析内容 → 设置 source_type=filesystem → 构建索引

新流程：扫描文件 → 检查 .xyddjson 文件
                         ↓
                    存在元数据文件?
                       / \
                      是   否
                     /     \
           解析 .xyddjson   source_type=filesystem
                     \          /
                      \        /
                       ↓      ↓
                  source_type=dingding
                       ↓
                  构建索引（包含source_url）
```

## 5. 详细设计

### 5.1 .xyddjson 格式规范

钉钉导出工具会为每个导出的文档生成一个同名元数据文件。

**文件结构示例**：

```json
{
  "version": "1.0.0",
  "source": "dingtalk-docs",
  "exportTool": "xiaoyaosearch-dingding-export",
  "exportToolVersion": "1.0.0",
  "exportTime": "2026-04-08T15:30:00.000Z",
  "file": {
    "fileName": "项目报告.docx",
    "originalName": "项目报告",
    "dentryUuid": "A1B2C3D4E5F6G7H8",
    "dentryKey": "key_abc123",
    "docKey": "doc_xyz789",
    "url": "https://alidocs.dingtalk.com/i/nodes/A1B2C3D4E5F6G7H8",
    "fileSize": 24576,
    "extension": "adoc",
    "exportFormat": ".docx",
    "contentType": "alidoc"
  }
}
```

**字段说明表**：

| 字段路径 | 类型 | 必填 | 说明 | 示例值 |
|---------|------|------|------|--------|
| `version` | string | 是 | 元数据格式版本 | `"1.0.0"` |
| `source` | string | 是 | 数据来源标识 | `"dingtalk-docs"` |
| `exportTool` | string | 是 | 导出工具名称 | `"xiaoyaosearch-dingding-export"` |
| `exportToolVersion` | string | 是 | 导出工具版本 | `"1.0.0"` |
| `exportTime` | string | 是 | 导出时间（ISO 8601） | `"2026-04-08T15:30:00.000Z"` |
| `file.fileName` | string | 是 | 导出后的文件名 | `"项目报告.docx"` |
| `file.originalName` | string | 是 | 钉钉文档原始名称 | `"项目报告"` |
| `file.dentryUuid` | string | 是 | 钉钉文档唯一ID | `"A1B2C3D4E5F6G7H8"` |
| `file.dentryKey` | string | 否 | 钉钉文档密钥 | `"key_abc123"` |
| `file.docKey` | string | 否 | 文档密钥 | `"doc_xyz789"` |
| `file.url` | string | 是 | 钉钉文档访问地址 | `"https://alidocs.dingtalk.com/i/nodes/..."` |
| `file.fileSize` | int | 否 | 文件大小（字节） | `24576` |
| `file.extension` | string | 否 | 原始扩展名 | `"adoc"` |
| `file.exportFormat` | string | 否 | 导出格式 | `".docx"` |
| `file.contentType` | string | 否 | 内容类型 | `"alidoc"` |

**文件命名规则**：
- 格式：`{原文件名}.{扩展名}.xyddjson`
- 示例：
  - `项目报告.docx` → `项目报告.docx.xyddjson`
  - `需求文档.pdf` → `需求文档.pdf.xyddjson`

**目录结构示例**：
```
D:\我的文档\钉钉导出\
├── 工作文档\
│   ├── 项目文档\
│   │   ├── 需求文档.docx
│   │   ├── 需求文档.docx.xyddjson      ← 元数据文件
│   │   ├── 技术方案.docx
│   │   └── 技术方案.docx.xyddjson      ← 元数据文件
│   └── 会议纪要\
│       ├── 周会记录.docx
│       ├── 周会记录.docx.xyddjson
│       ├── 产品讨论.pdf
│       └── 产品讨论.pdf.xyddjson
└── 市场文档\
    ├── 竞品分析.pdf
    └── 竞品分析.pdf.xyddjson
```

### 5.2 后端设计

#### 插件实现

**插件位置**：`backend/data/plugins/datasource/dingding/plugin.py`

**核心方法**：

```python
from pathlib import Path
import json
from typing import Dict, Any, Optional
from backend.app.plugins.interface.datasource import DataSourcePlugin


class DingdingDataSource(DataSourcePlugin):
    """钉钉文档数据源插件"""

    @classmethod
    def get_metadata(cls) -> Dict[str, Any]:
        return {
            "id": "dingding",
            "name": "钉钉文档",
            "version": "1.0.0",
            "type": "datasource",
            "datasource_type": "dingding",
            "description": "识别钉钉导出工具生成的 .xyddjson 元数据文件"
        }

    def get_file_source_info(self, file_path: str, content: str = None) -> Dict[str, Any]:
        """从钉钉导出的文档中提取元数据"""
        # 1. 检查是否存在同名 .xyddjson 文件
        # 2. 读取并解析 JSON 元数据
        # 3. 提取 dentryUuid、url、exportTime 等字段
        # 4. 返回标准化的数据源信息
        metadata = self._read_metadata_file(file_path)

        if not metadata:
            return {
                "source_type": None,
                "source_url": None,
                "source_id": None
            }

        return {
            "source_type": "dingding",
            "source_url": metadata["file"].get("url"),
            "source_id": metadata["file"].get("dentryUuid"),
            "export_time": metadata.get("exportTime"),
            "original_name": metadata["file"].get("originalName")
        }

    def _read_metadata_file(self, file_path: str) -> Optional[Dict[str, Any]]:
        """读取 .xyddjson 元数据文件"""
        metadata_path = Path(str(file_path) + '.xyddjson')

        if not metadata_path.exists():
            return None

        try:
            with open(metadata_path, 'r', encoding='utf-8') as f:
                metadata = json.load(f)

            # 验证元数据格式
            if metadata.get('source') != 'dingtalk-docs':
                return None

            return metadata
        except Exception as e:
            logger.warning(f"读取 .xyddjson 失败: {e}")
            return None
```

#### 元数据解析逻辑

**解析流程**：
1. 检查文件是否存在同名 `.xyddjson` 文件
2. 读取并解析 JSON 元数据
3. 验证 `source` 字段是否为 `dingtalk-docs`
4. 提取关键字段：`dentryUuid`、`url`、`exportTime`
5. 返回标准化的数据源信息

**错误处理**：
- 元数据文件不存在：返回 `None`，文件被视为本地文件
- JSON 解析失败：记录警告日志，返回 `None`
- 格式验证失败：返回 `None`

#### 文件过滤规则

在文件扫描服务中添加 `.xyddjson` 文件过滤：

```python
# 在文件扫描服务中过滤 .xyddjson 文件
IGNORED_PATTERNS = [
    '*.xyddjson',  # 钉钉元数据文件
    '*.tmp',
    '*.bak',
    # ... 其他模式
]
```

#### 业务逻辑变更

**文件扫描服务修改**：

```python
async def scan_file(self, file_path: str) -> FileInfo:
    """扫描单个文件"""
    # 检查钉钉元数据
    dingding_plugin = self._plugin_manager.get_plugin("dingding")
    source_info = dingding_plugin.get_file_source_info(file_path, None)

    # 设置数据源信息
    if source_info.get("source_type") == "dingding":
        file_info.source_type = "dingding"
        file_info.source_url = source_info.get("source_url")
        file_info.source_id = source_info.get("source_id")
    else:
        file_info.source_type = "filesystem"

    return file_info
```

**性能优化设计**：

| 优化点 | 实现方案 | 预期效果 |
|-------|---------|---------|
| 文件读取优化 | 先检查文件是否存在再读取 | 减少不必要的IO操作 |
| 缓存机制 | 缓存已解析的元数据 | 重复扫描时直接使用缓存 |
| 并行处理 | 批量并行读取元数据文件 | 提升扫描速度 |
| 错误容忍 | 元数据解析失败不影响原文件 | 保证鲁棒性 |

### 18.3 FAQ

**Q1: 如何使用钉钉文档搜索？**
A:
1. 安装钉钉导出工具（浏览器插件）
2. 使用导出工具导出钉钉文档
3. 将导出的文件放到小遥搜索的扫描目录
4. 系统会自动识别并建立索引

**Q2: .xyddjson 文件是什么？**
A: .xyddjson 是钉钉导出工具自动生成的元数据文件，包含文档的原始信息（URL、ID、导出时间等），用于小遥搜索识别文档来源。

**Q3: 可以删除 .xyddjson 文件吗？**
A: 可以删除，但删除后文档将无法被识别为钉钉数据源，会被当作普通本地文件处理。

**Q4: 钉钉文档搜索与飞书文档搜索有什么区别？**
A:
- 飞书：解析文件内容末尾的元数据块
- 钉钉：解析独立的 .xyddjson JSON文件
- 钉钉方案性能更好、可靠性更高、可扩展性更强

**Q5: .xyddjson 文件会出现在搜索结果中吗？**
A: 不会。系统会自动过滤 .xyddjson 文件，它们仅用于元数据提取，不会出现在搜索结果中。

**Q6: 钉钉原文链接会过期吗？**
A: 如果您有钉钉文档的访问权限，原文链接可以正常打开。链接本身不会过期，但文档可能被删除或权限变更。

**Q7: 为什么选择独立的元数据文件而不是嵌入文件内容？**
A: 独立元数据文件有以下优势：
1. 结构化JSON格式，解析更可靠
2. 无需读取文件内容，性能更好
3. 元数据与文件分离，易于维护
4. 可扩展到其他平台，通用性强

**Q8: 如何升级元数据格式？**
A: 元数据文件包含 `version` 字段，未来升级时会支持多版本格式兼容，旧格式仍可正常使用。
