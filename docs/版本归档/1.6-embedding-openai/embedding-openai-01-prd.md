## 1. 特性概述

### 1.1 定位（一句话）
为小遥搜索添加云端嵌入模型调用能力，用户可在"嵌入模型"配置选项中选择使用本地或云端API（OpenAI/DeepSeek/阿里云等），两者互斥。

### 1.2 背景与动机

**用户痛点**：
- 痛点1：本地模型需要较高硬件配置（建议8GB+显存），低配设备运行困难
- 痛点2：本地模型对英文、日文等非中文语言的嵌入质量不如专业云端模型
- 痛点3：部分用户需要更高质量的嵌入向量以提升搜索准确度，愿意为此付费
- 痛点4：云端嵌入模型更新迭代更快，能获得更好的语义理解能力

**业务价值**：
- 降低使用门槛：低配设备用户可通过云端API获得高质量搜索体验
- 提升搜索质量：云端模型（如OpenAI text-embedding-3）在语义理解上更优
- 扩展用户群体：吸引对搜索质量有高要求的专业用户
- 建立商业模式：为未来推出云服务订阅制奠定基础

**技术必要性**：
- 项目已具备完整的AI模型配置架构，支持local/cloud双provider模式
- 已有OpenAI兼容LLM服务实现，可作为技术参考
- 云端嵌入调用是现有架构的自然延伸，技术风险低

### 1.3 目标用户

**主要用户**：
- 专业内容创作者：需要更精准的语义搜索
- 多语言用户：需要高质量的非中文嵌入
- 低配设备用户：无法流畅运行本地模型
- 企业用户：愿意为高质量服务付费

**使用场景**：
- 场景1：用户在搜索英文文档时，切换到云端模型获得更准确的语义理解
- 场景2：低配笔记本用户使用云端API替代本地模型，释放系统资源
- 场景3：用户在搜索学术资料时，使用OpenAI text-embedding-3提升召回准确度

### 1.4 与主版本的关系

**依赖关系**：
- 依赖主PRD中的AI模型配置模块（ai_models表）
- 依赖主PRD中的向量搜索引擎（Faiss索引）
- 依赖OpenAI兼容大模型服务特性的配置结构设计

**影响范围**：
- 影响：AI模型管理服务、嵌入模型配置选项卡
- 不影响：现有的Faiss索引结构（统一1024维）
- 互斥关系：云端和本地嵌入模型互斥，用户只能选择一种

## 3. 用户故事（增量场景）

> **说明**：只记录新增场景或重大变更的场景

**场景1：专业用户配置云端嵌入模型**

- 作为一名专业内容创作者
- 我想要在嵌入模型配置选项卡中选择使用云端API
- 以便于在搜索英文资料时获得更准确的语义理解

**验收标准**：
- [ ] 可以在嵌入模型配置选项卡中选择"云端API"
- [ ] 可以输入API Key、端点地址、模型名称
- [ ] 配置保存后可以进行连接测试
- [ ] 测试通过后系统使用云端嵌入模型

**场景2：低配设备用户切换到云端模型**

- 作为一名使用低配笔记本的用户
- 我想要在嵌入模型配置选项卡中切换到云端API
- 以便于释放本地计算资源，提升系统响应速度

**验收标准**：
- [ ] 可以在嵌入模型配置选项卡中从本地切换到云端
- [ ] 切换后本地模型被卸载，释放资源
- [ ] 搜索功能正常工作，使用云端API
- [ ] 搜索结果质量与本地模型相当或更好

**场景3：本地模型用户切换回云端**

- 作为一名云端API用户
- 我想要切换回本地模型
- 以便于节省API费用，或在无网络环境下使用

**验收标准**：
- [ ] 可以在嵌入模型配置选项卡中选择"本地"
- [ ] 切换后云端API不再被调用
- [ ] 系统加载本地模型
- [ ] 搜索功能正常工作

## 4. 核心流程（增量部分）

### 4.1 修改流程：配置嵌入模型（本地/云端互斥）

```
用户打开配置界面
    ↓
选择"AI模型配置" → "嵌入模型"选项卡
    ↓
选择嵌入模型来源（单选）：
  ○ 本地（默认）
  ○ 云端API
    ↓
[如果选择云端API]
    ↓
填写云端配置信息：
  - 提供商（下拉选择：OpenAI/DeepSeek/阿里云/自定义）
  - API Key（加密存储）
  - 端点地址（自动填充，可修改）
  - 模型名称（自动填充推荐模型，可修改）
  - 维度处理模式（截断/投影/原生）
    ↓
点击"测试连接"
    ↓
后端调用一次embedding API验证配置
    ↓
测试通过 → 保存配置，系统切换到云端嵌入模型
测试失败 → 显示错误信息，保持当前配置
    ↓
[如果选择本地]
    ↓
保存配置，系统切换到本地嵌入模型
```

### 4.2 修改流程：系统启动时的嵌入模型加载

```
系统启动
    ↓
读取嵌入模型配置（provider字段）
    ↓
provider='local' → 加载本地模型
provider='cloud' → 初始化云端API客户端
    ↓
模型就绪，等待搜索请求
```

## 7. 技术方案

### 7.1 技术选型

| 技术/框架 | 用途 | 选择理由 | 替代方案 |
|----------|------|---------|---------|
| aiohttp | HTTP客户端 | 异步高性能，与OpenAI LLM服务一致 | httpx、requests |
| numpy | 向量处理 | 高效数值计算，与现有代码一致 | TensorFlow、PyTorch |
| tenacity | 重试机制 | 声明式重试，代码简洁 | 手动重试逻辑 |
| BaseAIModel | 基类 | 继承现有架构，保证一致性 | 新建独立服务 |

### 7.2 架构变更

**前端架构**：
- 修改AI模型配置页面，在"嵌入模型"选项卡内添加本地/云端互斥选择

**后端架构**：
```
API层
  ↓
AI模型配置服务 (ConfigService) → 保存嵌入模型配置（local/cloud互斥）
  ↓
AI模型管理服务 (AIModelService) → 根据配置加载对应模型
  ↓
模型管理器 (ModelManager)
  ↓
服务实现层（互斥，同时只有一个活跃）
  ├── BaseAIModel (ABC)
  ├── BGEEmbeddingService (provider='local') → 本地
  └── OpenAIEmbeddingService (provider='cloud') ← 新增，云端API
  ↓
搜索服务 (SearchService) → 使用当前活跃的嵌入模型
```

**数据流**：
```
系统启动
    ↓
读取嵌入模型配置
    ↓
provider='local' → 加载BGEEmbeddingService
provider='cloud' → 加载OpenAIEmbeddingService
    ↓
用户搜索请求
    ↓
使用当前活跃的嵌入服务获取向量
    ↓
获取嵌入向量（1024维）
    ↓
传入Faiss搜索引擎
    ↓
返回搜索结果
```

### 18.3 FAQ

**Q1: 云端嵌入模型和本地有什么区别？**
A: 云端模型通常提供更高的语义理解质量和多语言支持，但需要网络连接和API费用。本地模型免费且离线可用，但需要较高硬件配置。两者互斥，用户只能选择一种使用。

**Q2: 如何在本地和云端嵌入模型之间切换？**
A: 在"AI模型配置"页面的"嵌入模型"选项卡中，选择"本地"或"云端API"，保存后系统会自动切换。切换模型时需要重建索引，因为不同模型的向量空间不兼容。

**Q3: API调用失败怎么办？**
A: 系统会自动重试3次，如果仍然失败，建议切换回本地模型使用。

**Q5: 云端和本地模型可以同时使用吗？**
A: 不可以。云端和本地嵌入模型是互斥的，系统同时只会加载一种嵌入模型。
